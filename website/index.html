Why veyyon

Six ways the harness earns its keep.

Every part of veyyon exists to get more real work out of the model you already run. What each one does, and why it matters.

01

Edits that land

Hashline anchors every edit to the content of your file, not to line numbers, and each hunk is verified before it applies. When a patch drifts, the harness repairs it in place instead of corrupting the file or spending a turn on a retry. Fewer malformed diffs, fewer wasted turns — and a smaller model stops tripping over its own output.

02

A plan before anything mutates

Plan mode gates every tool to read-only work while the agent grounds itself in your code and lays out what it intends to do. You approve the shape of the work first, then it executes. You always see the plan before the diff — nothing changes on a guess.

03

Three model slots, each chosen by you

The model you talk to, the model your subagents run, and the model that compacts context are three separate settings, in the open. There's no default slot quietly blending them: switch your interactive model and the background slots stay exactly where you put them.

04

Long sessions that keep the thread

Compaction is three knobs: a threshold (with a per-model default), a type (handoff or snap), and a model — not a wall of settings. Memory is durable and opt-in. Sessions fork, clone, and label with /tree, so exploring a path never costs you the trunk.

05

A boundary the OS enforces

Approvals ask; the sandbox enforces. Landlock, seccomp, and Seatbelt back read-only, workspace-write, and full-access tiers, failing closed by default — so the boundary holds even when nobody is watching.

06

Every agent in view

The cockpit shows your whole fleet live: a model per subagent, status as it changes, and a drill-in for any thread. Multi-agent work becomes something you watch and steer, not a blind fan-out you audit after the fact.

How it works

A typed agent loop over native hot paths.

No magic in the middle: a disciplined loop, a fast core, and interfaces you can build on.

The loop

Ground, propose, act, verify.

The agent reads your repo before it plans, plans before it mutates, and runs your tests when it is done. A review pass can check its own changes, on a model you choose, before anything lands.

The stack

TypeScript and Bun, Rust where it counts.

The agent loop and the TUI are TypeScript on Bun. Rust owns the hot paths — repo walking, grep, the PTY, and the edit engine — so the interface stays responsive mid-task.

Open seams

Built to be extended.

MCP servers, SKILL.md skills, and lifecycle hooks plug into every turn. A real headless browser and multi-provider web search are built in, and headless exec runs the same agent in CI.

The model layer

Any provider. Your keys. Explicit slots.

Anthropic, OpenAI, Google, Groq, xAI, OpenRouter, Mistral, and Bedrock — plus Ollama, LM Studio, and llama.cpp found automatically, keyless. Your key talks to your provider directly.

  • Interactive model — the one you talk to. Switch live with /model, or launch with --model.
  • Subagent model — one setting; every background agent runs on it. Keep it cheap on purpose.
  • Compaction model — one setting; context shrinks on a model that fits the job and the bill.
  • Roles, optional — named assignments for work types, scoped per profile, edited in settings — never in the picker.
# ~/.veyyon/config.yml
model: openai/gpt-5            # interactive — or /model live
subagent:
  model: deepseek/deepseek-chat
compaction:
  model: openai/gpt-5-mini

About

What Veyyon is.

Veyyon is a fork of oh-my-pi (MIT) by Can Boluk, credited on every page. It's a coding agent you run in your own terminal, on your own repo, with whatever model you configure. It doesn't train models or rank them; it's the harness around the model — prompts, editing, tools, context — and that's the part it works on. Everything is configurable in the handbook, and the changelog is generated from the real CHANGELOG and the published GitHub releases.

Quickstart

Running in a minute.

Install

curl -fsSL https://get.veyyon.dev | sh — or grab a release binary from GitHub. PATH, completions, and a doctor self-test are all handled.

Sign in

/login <provider> or set the provider env var. Any OpenAI-compatible endpoint works; local engines are found automatically and need no key.

Build

vey in any repo, /model to pick your model, and start with a plan.

Get started

One command away.

Bring a key from any provider — or none at all, if your models run local. The first session takes a minute.

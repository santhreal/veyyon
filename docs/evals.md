# Veyyon Evaluation System Architecture & Workflow

This document describes the design, codebase organization, use cases, and execution workflows for Veyyon's standing evaluation system (`packages/deepswe-bench`).

## 1. Codebase & Package Architecture

The evaluation system is housed under `packages/deepswe-bench/`:

```text
packages/deepswe-bench/
├── run.ts                 <-- Main CLI runner & report generator
├── README.md              <-- Package workflow documentation
├── package.json           <-- Node/Bun workspace configuration
├── arms/                  <-- Experiment arm overlays & system prompt candidates
│   ├── baseline.yml       <-- Standard production flag overlay
│   ├── candidate-v6.yml   <-- Experiment arm flag overlay
│   └── candidate-v6.prompt.md <-- Candidate system prompt template
├── tasks/                 <-- Curated task suites
│   ├── smoke.txt          <-- 1-task fast smoke test (ytt-jsonpath-query-api)
│   ├── pilot-10.txt       <-- 10 hardest long-horizon tasks
│   ├── argot-10.txt       <-- 10 high-mass token tasks
│   └── diverse-20.txt     <-- 20 multi-repo tasks across Go, TS, and Python
├── pier_agent/            <-- Pier container agent wrapper
│   ├── veyyon_agent.py    <-- Uploads binary/auth/arm into Pier Docker containers
│   └── oneshot_prompt.md.j2 <-- Harbor task instruction jinja template
└── assets/                <-- Staged binary (dist/vey) and seeded auth DB
```

---

## 2. Supported Evaluation Use Cases

The evaluation harness supports four distinct experimentation workflows:

### A. System Prompt Candidates
- Create `arms/<arm_name>.yml` (config flags) and `arms/<arm_name>.prompt.md` (prompt template).
- `run.ts` automatically stages the prompt file into the Docker container and passes `--system-prompt` to `vey`.

### B. Feature Flags & Config Overlays
- Toggle settings in `arms/<arm_name>.yml` (e.g. `argot.enabled: true`, `discovery.importForeignConfig: true`, `ttsr.builtinRules: false`).
- Compare baseline vs feature overlay across tasks.

### C. Multi-Provider & Model Comparisons
- Run the benchmark with `--model <provider/model-id>` (e.g. `google-antigravity/gemini-2.5-flash`, `google-antigravity/claude-opus-4-6`, `anthropic/claude-3-7-sonnet`).

### D. Multi-Language Task Suites
- Evaluate performance across Python, Go, and TypeScript repositories (`tasks/diverse-20.txt`).

## 3. Repeatable, Verifiable Output Structure

Every run creates an isolated, timestamped output directory under `packages/deepswe-bench/runs/<label>/`:

```text
runs/<label>/
├── report.md         <-- Side-by-side Markdown comparison table (Reward, Tokens, Cost, Tool Calls)
├── results.json      <-- Machine-readable JSON metrics dataset
├── assets/           <-- Uploaded sha256-hashed binary, auth DB, and arm prompts
└── jobs/
    └── <arm>__<task>/
        ├── agent/
        │   ├── veyyon.txt    <-- Agent stdout/stderr log
        │   └── sessions/
        │       └── *.jsonl   <-- Verbatim JSONL session traces of every turn & tool call
        ├── verifier/
        │   └── test.log      <-- Output from held-out test verifiers
        └── patch.diff        <-- Exact git patch generated by the agent
```

---

## 4. Re-Aggregation & Off-Line Trace Analysis

To re-calculate metrics or rebuild `report.md` after updating accounting logic without re-running LLM trials:
```bash
bun packages/deepswe-bench/run.ts --reaggregate packages/deepswe-bench/runs/<label>
```

---

## 5. Architectural Safeguards

1. **Zero Profile Impact:** All trials execute in Pier/Docker sandboxes with read-only credential copies. Host profile files (`~/.veyyon/profiles/`) are never touched.
2. **Auto-Build & Auto-Auth:** `run.ts` automatically detects stale binaries in `dist/vey` and auto-rebuilds them. It also auto-seeds `assets/auth-agent.db` from your active session credentials.
3. **Container Collision Protection:** `run.ts` automatically prunes orphaned docker networks and retries on container startup collisions.
4. **Prompt Cache Protection:** System prompt headers (`<workstation>`) remain static mid-session prior to context compaction to avoid 100% cache-miss token inflation.

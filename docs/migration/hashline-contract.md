# hashline tokenizer/normalize contract

This document freezes the public boundary of the first-wave port candidate:
the `splitHashlineLines` / `parseLid` tokenizer entry points and the four
`normalize` functions. A port reimplements exactly what is written here. If
the code and this document disagree, that is a defect in one of them; fix it
in a reviewed change that updates both, never by silently following the code.

No module is ported before its contract exists. This is the rule of record
for the migration effort.

The machine-readable twin of this document is `manifest.json` in the same
directory. A test (`packages/hashline/test/contract-manifest.test.ts`)
asserts that every function listed there exists and is exported, so the
contract cannot drift from the source without failing CI.

## splitHashlineLines(text: string) -> string[]

Splits `text` into lines the way the hashline patcher counts them.

- The empty string is one empty line: `[""]`.
- LF (`\n`) terminates a line. A CR immediately before a terminating LF is
  trimmed from that line (CRLF handling). A lone CR anywhere else is content.
- Terminator-style semantics: a trailing terminator adds no phantom line
  (`"a\n"` is `["a"]`). Consequence: a document whose final line is empty is
  indistinguishable from one with a trailing terminator; the lossless
  round-trip `split(x).join("\n") === x` holds exactly when `x` does not end
  with a terminator.
- A trailing lone CR at end of input is trimmed from the final line.
- No other code point splits: U+2028, U+0085, and U+000B/U+000C are content.
- The result never has zero elements, and no element contains LF.
- Errors: none. Total on all strings.

## parseLid(raw: string, lineNum: number) -> Anchor

Parses a bare line-number anchor.

- Accepts: optional ASCII whitespace padding (space, TAB, or the C0 range
  TAB..CR) around one decimal integer with a non-zero first digit.
  Returns `{ line: <number> }`.
- The integer is accumulated in a JS number; every positive integer up to
  `Number.MAX_SAFE_INTEGER` round-trips its canonical decimal rendering.
- Rejects (throws `Error`): empty/whitespace-only input, `0`, any leading
  zero, signs, decimal points, exponents, hex, grouping separators, any
  trailing or leading non-whitespace garbage, two numbers, and every
  non-ASCII digit (fullwidth `５`, Arabic-Indic `٥`, NBSP is not whitespace).
- Error message: starts with `line <lineNum>: expected a line number`,
  embeds the offending input via `JSON.stringify`, and names the
  file-version-binding syntax. The `<lineNum>` argument appears verbatim, so
  callers must pass the 1-based hashline line number.
- Nondeterminism: none.

## detectLineEnding(content: string) -> "\r\n" | "\n"

First-encounter detection.

- If the content has no LF at all, the result is `"\n"` (a lone CR is not an
  ending).
- Otherwise the result is `"\r\n"` exactly when the first CRLF occurs before
  the first LF, else `"\n"`.

## normalizeToLF(text: string) -> string

Replaces every CRLF and every lone CR with LF (regex `/\r\n?/g`).

- Idempotent; the result contains no CR byte.
- Byte pair CR LF is ONE ending even when the CR and LF were conceptually
  written as two separate boundaries (frozen conformance vector
  `"\n\r\n\n"` -> `"\n\n\n"`).

## restoreLineEndings(text: string, ending: LineEnding) -> string

- With ending `"\n"`: identity.
- With ending `"\r\n"`: every LF becomes CRLF. Pre-existing CR bytes are NOT
  protected: text already containing CRLF doubles its CR (documented and
  recorded in the corpus). Callers pass LF-normalized text.
- `restoreLineEndings(normalizeToLF(x), detectLineEnding(x)) === x` for any
  document written in a single ending style.

## stripBom(content: string) -> { bom: string, text: string }

- Splits one leading U+FEFF into `bom`, everything else into `text`;
  `bom + text === content` always.
- Only the FIRST leading BOM is stripped; a second BOM stays in `text`.
  Interior BOMs are content.

## Determinism

None of these functions read the clock, RNG, environment, locale, or
filesystem, and none depend on collection iteration order. The package-level
lock is `packages/hashline/test/determinism-lock.test.ts`; the only allowed
clock read in the package (`snapshots.ts` `recordedAt`) never reaches these
outputs.

## Verification stack

- Conformance corpus: `packages/hashline/test/conformance/vectors/` (153
  vectors), regenerated by `bun scripts/record-conformance.ts hashline`.
- Properties: `packages/hashline/test/property/` (10 invariants x 10k runs).
- Perf baseline: `packages/hashline/bench/hot-paths.bench.ts` (machine-relative
  `baseline.json`; a port must meet or beat it on the reference box).
- Boundary coverage floor: `packages/hashline/scripts/coverage-floor.ts`
  (`bun run coverage:boundary`) proves the black-box suites alone drive the
  contracted sources past per-file floors.
- Golden patcher outputs: `packages/hashline/test/golden/` pins the full
  operator-visible result of end-to-end patch applications (section results,
  warnings, error wording, line-ending restoration, final filesystem bytes).
  Update with `UPDATE_GOLDEN=1 bun test test/golden` and review the diff.
- Format spec for the vector files: `conformance-format.md`.
